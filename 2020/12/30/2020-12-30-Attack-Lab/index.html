    背景

    Attack lab是CSAPP课程的第三个实验，延续了第二个实验Bomb，继续在关注x86_64的汇编语言。不同之处在于，Bomb lab只需要能够读懂汇编代码即可，但Attack lab在理解汇编的基础上要有一定写汇编的能力。

    栈溢出

    一些函数中会声明数组来接受用户的输入，这样的数组如果位于栈中，则用户可能通过恶意地输入大于数组大小的输入，从而修改栈中内容来进行代码注入(Code Injection)。以下图为例

    图中数组buf的大小为8个字节，如果用户输入大于8个字节，则栈溢出。如果用户输入大于24个字节，则echo函数返回时，%rsp+24里面的内容已经被修改。特殊地，如果此时%rsp+24中已经被修改为某段代码的地址，则程序会开始执行用户所希望的其它代码。

    栈溢出的保护

    事实上，上面的这种代码注入方式在现代的程序中其实已经难以适用了，因为下面的栈溢出的保护方法已经被编译器广泛地使用了。常见的栈保护方法有

    1. Stack Randomization (栈地址的随机化)

      代码注入时，经常会需要知道栈中的某些地址。因此，编译器会让栈的起始地址随机化，这样两次运行会有不同的地址，提高了代码注入的难度。

    2. Limiting Executable Code Region （限制可运行代码的区域）

      通常，正常编译的程序的指令位于指令区(.text region)。因此，编译器可以考虑禁止栈上的内容可执行。

    3. Stack Corruption Detection (检测)

      现代的编译器会buffer之上，放置一个canary（金丝雀） value。函数返回时，会检测canary value是否被修改。一旦发现修改，则直接停止执行。如下图所示。

    当然，本次实验中这些保护机制被部分或全部关闭，否则我们很难进行攻击。

    Attack lab

    这个实验分为两个部分

    1. Code Injection Attack: 这部分中，三个保护机制都被关闭，我们可以直接溢出buffer修改栈上的内容，从而注入我们的代码
    2. Return-Oriented Attack: 这部分中栈的地址有随机性，因此我们不能直接使用某些关键的地址。其次，栈中内容不能被执行，因此我们无法直接注入我们的恶意代码。具体如何做，见下文。

    返回导向编程 Return oriented programming

    上面的代码注入有两个前提条件，第一，栈的地址不能是随机的；第二，栈上的代码可以执行。那么，如果这两个条件不能满足呢？Return oriented programming的思路是，既然我插进去的代码无法运行，那么只好重复利用程序字节码中本来就位于.test区的代码。

    返回导向的意思是我们只打算重复利用函数返回之前的代码，这样把一系列执行之后就立即返回的代码排列在栈里，就可能达到我们想要的效果。这里需要指出的是，由于我们只是从程序本来的字节码中摘抄，所以执行的效果可能与他们所处函数中的作用完全不相干。举例来说，如果有一个函数的结尾汇编是如下的形式，
